# Архитектура системы QGroundControl

В данном документе представлен высокоуровневый архитектурный анализ наземной станции (GCS) QGroundControl (QGC).

---

## 1. Общий архитектурный стиль

QGroundControl представляет собой **кроссплатформенный монолитный "толстый клиент"** (Thick Client).
- **Слоевая архитектура:** Четкое разделение между транспортным слоем, бизнес-логикой и слоем визуализации.
- **Технологический ландшафт:** Приложение построено на фреймворке Qt. Вся бизнес-логика (работа с потоками, портами, MAVLink, хранение параметров) написана на C++. Весь интерфейс реализован на QML (JavaScript).

---

## 2. Основные архитектурные паттерны

- **MVC / MVVM с использованием кастомной Fact System:** Роль ViewModel играют C++ классы (`Fact`, `FactGroup`), выступающие связующим звеном между "сырыми" параметрами и реактивным QML-интерфейсом.
- **Publisher-Subscriber (через Signals & Slots):** Глубокая интеграция с реактивной моделью Qt. Данные, приходящие от БПЛА, порождают события (сигналы), автоматически уведомляя UI о необходимости перерисовки без прямого Coupling-а.
- **Паттерн "Фасад" / "Стратегия" (Firmware Plugin):** Набор абстрактных классов (`FirmwarePlugin`), которые скрывают разницу в реализациях диалектов MAVLink у разных автопилотов (PX4, ArduPilot) от общего пользовательского интерфейса.
- **Конечные автоматы (State Machine):** Используются для управления сложными многоступенчатыми процессами, например, для инициализации подключения (`InitialConnectStateMachine`) или управления этапами обновления прошивки.

---

## 3. Ключевые модули и их роли

| Модуль | Расположение | Зона ответственности |
|---|---|---|
| **Comms (LinkManager)** | `src/Comms` | Сетевой/физический транспорт. Поддержка Serial (USB), UDP, TCP. Чтение и запись сырых байтов без анализа. |
| **MAVLink** | `src/MAVLink` | Десериализация сырых байтов в пакеты MAVLink. Маршрутизация сообщений (кому и от кого пришел пакет). |
| **Vehicle** | `src/Vehicle` | Цифровой двойник дрона. Хранит координаты, текущие режимы, статус вооружения (armed) и объединяет все системы (датчики, батарею). |
| **FactSystem** | `src/FactSystem` | Абстракция переменных. Описывает любой параметр в виде `Fact` (содержит значение, мин/макс пределы, единицы измерения, дефолт). |
| **FirmwarePlugin** | `src/FirmwarePlugin` | Изоляция специфики прошивки БПЛА. Преобразует универсальные UI команды (например. "Взлёт") в специфичную команду автопилота. |
| **MissionManager** | `src/MissionManager` | Механизмы планирования миссий, формирование путевых точек (Waypoints), точек возврата (Rally) и "заборов" (GeoFence). Синхронизация миссии с БПЛА. |
| **VideoManager** | `src/VideoManager` | Стриминг видео с борта. Контекстный пайплайн на основе GStreamer (RTSP, UDP). |
| **UI** | `src/ui`, `src/QmlControls` | Компоненты отрисовки интерфейсов отображения (`FlyView`) и планирования (`PlanView`). |

## 5. Зависимости и граница ответственности модулей

- **Транспорт (Comms) vs Протокол (MAVLink):** Comms не знает ничего о структуре сообщений, только о скорости порта и TCP/UDP сокете. MAVLink знает поля сообщений, но не понимает их бизнес-влияния на дрон.
- **Модель (Vehicle) vs UI:** Vehicle ничего не знает о том, как его свойства отображаются на экране. Он меняет внутренние переменные и делает `emit`. UI не обрабатывает пакеты: он лишь "слушает" сигналы от Vehicle.
- **Vehicle vs FirmwarePlugin:** Vehicle выступает "мозгом" архитектуры приложения, но любую команду зависящую от автопилота делегирует в FirmwarePlugin: `vehicle->firmwarePlugin()->someAction()`.

---

## 6. Как UI связан с backend логикой

Архитектура строго разделена по принципу **C++ Backing / QML Fronting**.

1. Компоненты UI (написанные на QML) не делают запросы/расчеты сложной физики. Они обращаются к C++ объектам через внедренный в контекст QML мета-объект `QGroundControl`.
2. В классе `Vehicle` или `Fact` определены макросы `Q_PROPERTY(bool armed READ armed WRITE setArmed NOTIFY armedChanged)`.
3. Когда по радио приходит MAVLink пакет, C++ бэкенд вычисляет новое состояние, изменяет поле и вызывает сигнал `emit armedChanged()`.
4. Движок Qt Quick (QML) автоматически, за миллисекунды, перехватывает сигнал и перерисовывает те части интерфейса, в которых использовалась переменная `vehicle.armed` (без использования JavaScript таймеров или HTTP поллинга).

## 7. Как система масштабируется

Так как система является настольным (и мобильным) клиентом, здесь отсутствуют классические понятия распределенного (cloud-native) масштабирования, но: 

- **Горизонтальное масштабирование роя:** QGroundControl масштабируется до множества одновременно контролируемых беспилотников через паттерн `MultiVehicleManager`. Каждый дрон — это изолированный инстанс класса `Vehicle` со своей FactSystem и своим MAVLink-каналом. 
- **Архитектурная расширяемость (Extensibility):** С помощью плагинов (`FirmwarePlugin`, `AutoPilotPlugin`) можно внедрить поддержку нового типа дронов (например, проприетарного коммерческого БПЛА) без вмешательства в ядро связи или переписывания UI планировщика.
- **Ограничения:** Масштабирование упирается исключительно в пропускную способность радио-телеметрии и вычислительные ресурсы компьютера/планшета для рендеринга видео (GStreamer) и множества 3D объектов на карте (QtLocation).

# Анализ ключевых архитектурных решений QGroundControl

Документ описывает и анализирует основные инженерные компромиссы (Architecture Decision Records) и их влияние на проект QGC.

---

## 1. Концепция: Монолитное приложение на базе Qt

**Проблема:** Необходимо создать единую станцию управления БПЛА, которая могла бы работать на полевых ноутбуках (Windows, Linux, macOS) и мобильных пультах (Tablet, iOS, Android), одновременно обеспечивая доступ к "железу" (Serial порты, радиомодули) и высокопроизводительный рендеринг видео или 3D.

- **Плюсы:**
  - *Write once, run anywhere:* Единая кодовая база `C++ / QML` компилируется под все целевые платформы.
  - Нативный доступ к сетевому и аппаратному стеку через Qt Network и Qt SerialPort, что недостижимо в браузерных приложениях без прослоек.
  - Высокая графическая производительность за счет Qt Quick (OpenGL/Vulkan), что критично для отрисовки видео GStreamer и карт.
- **Минусы:**
  - Жесткий *Vendor Lock-in* на экосистему Qt.
  - Огромный размер итогового бинарного файла и потребление оперативной памяти из-за включения всех Qt библиотек.
- **Ограничения:** Сложность обновления фреймворка (например, болезненная миграция с Qt5 на Qt6).

## 2. Архитектурный паттерн: Управление состоянием через Fact System

**Проблема:** Современные автопилоты (PX4 / ArduPilot) экспортируют тысячи параметров. Жесткое кодирование переменных для каждого параметра в коде интерфейса привело бы к колоссальным объемам boilerplate-кода.

- **Плюсы:**
  - Полностью data-driven UI. Все интерфейсы (ползунки, чекбоксы) генерируются и валидируются автоматически на основе метаданных (`FactMetaData`), приходящих с контроллера либо из кэша.
  - Защита от дурака: `FactSystem` сама проверяет диапазон значений (`min`/`max`) перед отправкой на борт.
  - Идеальная совместимость с QML за счет биндинга `Q_PROPERTY`. 
- **Минусы:**
  - Отладка (Debugging) и навигация по коду усложнена: трудно отследить через стек вызовов, почему обновилась какая-то переменная (все скрыто под абстракцией `emit valueChanged`).
- **Ограничения:** Требует загрузки в память огромных словарей метаданных для каждого типа подключенного устройства.

## 3. Слой абстракции: Использование плагинов (FirmwarePlugin)

**Проблема:** Исторически сложилась конкуренция двух крупнейших систем БПЛА — ArduPilot и PX4. Они обе используют MAVLink, но реализация команд (диалекты, режимы полета) у них кардинально отличается.

- **Плюсы:**
  - Изоляция специфичной логики ("Предположение": если пользователь выбирает "Return To Launch", плагин `PX4` отправит одну последовательность MAV_CMD, а `APM` — другую, но кнопка в UI останется одной и той же).
  - Высокая масштабируемость системы для коммерческих вендоров: компании могут создать собственный закрытый плагин в директории `custom/` под свой кастомный дрон.
- **Минусы:**
  - Эффект "Наименьшего общего знаменателя". Любая новая функция автопилота должна пройти через абстракцию `FirmwarePlugin`, что усложняет использование уникальных возможностей конкретной прошивки.
- **Ограничения:** Ядро приложения обязано поддерживать легаси-режим (если какие-то методы в плагине не реализованы, необходимо fallback-поведение).

## 4. Архитектурный тип: Thick Client (Толстый клиент)

**Проблема:** В полевых условиях (в степи, в горах или зонах без связи) часто отсутствует интернет. Управление БПЛА критично к задержкам.

- **Плюсы:**
  - Система абсолютно автономна (за исключением загрузки онлайн-карт, для которых предусмотрен кэш SQLite).
  - Минимизация задержек (Latency): радио-модемы или USB кабель подключаются напрямую к GCS устройству, парсинг телеметрии происходит мгновенно на том же CPU.
- **Минусы:**
  - Сложность совместной (Collaborative) работы нескольких операторов (например, удаленный диспетчерский центр в другом городе "из коробки" напрямую не может наблюдать за таким клиентом, требуется настройка параллельного стриминга).
  - Вся тяжелая обработка (парсинг гигантских логов, работа с видео) ложится на батарею планшета в поле.
- **Ограничения:** Сложность "бесшовных обвнолений": пользователь обязан скачивать новые установщики `apk/exe` файлов для обновления софта.

## 5. Протоколы: MAVLink вместо REST API

**Проблема:** REST (HTTP) не подходит для работы в радиолокационном пространстве с высокой долей шума, потери пакетов и низкой пропускной способностью (радиомодемы на 915 МГц часто дают до 10-50 кбит/сек).

- **Плюсы:**
  - Высшая степень экономии битов. MAVLink использует плотную бинарную упаковку.
  - Потоковая передача данных (Pub/Sub концепция поверх UDP) естественна для телеметрии (`ATTITUDE`, `GPS`), где потеря одного кадра некритична, так как через 50мс придет новый.
- **Минусы:**
  - Отсутствие транзакционности. Нет механизма "запрос-ответ" как в классическом вебе (гарантированная доставка реализуется сложными конечными автоматами сверху протокола — например, протокол Mission/FTP).
  - Отсутствие REST у самого приложения: QGroundControl сложно интегрировать со сторонними Enterprise CRM или Dashboard/Grafana системами по стандартным веб-векторам.
- **Ограничения:** ("Предположение"): QGC замкнут в экосистеме MAVLink. Управление кастомным Payload-ом (специализированные датчики, лидары без MAVLink-обертки) требует костылей.

## 🛠 Потенциальные слабые места и технические риски системы

1. **Риск Single Event Loop (Модель блокировки UI)**  
   *("Предположение")*: Архитектура Qt подразумевает выполнение C++ логики (событий) и QML отрисовки в тесном цикле. Обработка гигантских объемов приходящих пакетов телеметрии от "роя" БПЛА (Swarm) может вызвать деградацию частоты кадров GUI на слабых Android-планшетах.
2. **Риски памяти (Memory Leaks)**  
   Использование C++ и ручного перевыделения моделей в QML (создание и удаление объектов `Vehicle` и сотен свойств при подключении-переподключении) может привести к утечкам. В `Vehicle:prepareDelete()` присутствуют сложные цепочки освобождения ресурсов.
3. **Безопасность (Security Risk)**  
   MAVLink традиционно был не зашифрован. Любой, кто подключится к порту 14550 в Wi-Fi сети устройства, потенциально может перехватывать телеметрию (в коде видно наличие поддержки подписанных MAVLink сообщений, `mavlinkSigning`, однако само приложение является огромным доверенным шлюзом).
4. **Сложность тестирования (Technical Debt Risk)**  
   Глубокая интеграция UI `QML` со сложными C++ моделями `FactSystem` крайне тяжела в End-to-End автотестировании (модульные тесты `QGC_UNITTEST_BUILD` покрывают бэкенд, но тест интерактивного UI на картах требует огромных затрат на QA-инфраструктуру).

# Динамика и жизненный цикл системы QGroundControl (Data Flow)

Документ описывает полный lifecycle системы, фокусируясь на потоках данных, смене состояний и синхронизации компонентов от момента запуска приложения до конца выполнения полетной миссии.

---

## 1. Startup приложения

**Сценарий:** Приложение запускается на клиентском устройстве (Desktop / Mobile).

- **Участвующие модули:** `QGCApplication`, `LinkManager`, `MultiVehicleManager`, `FirmwarePluginManager`, `UI (QQmlApplicationEngine)`.
- **Какие данные передаются:** Локальные конфигурации (допустимые скорости портов, сохраненные Autoconnect-линки) из `SettingsManager` считываются в память.
- **События-триггеры:** Системный вызов OS для старта приложения `main.cc -> app.init()`.
- **Изменяемые состояния:** Состояние системы: `Disconnected`. Мобильные интерфейсы ожидают коннекта.

**Step-by-step последовательность:**
1. Инициализация платформозависимых видео- и сетевых конвейеров (GStreamer, Qt Network).
2. `LinkManager` открывает порты в режиме прослушивания (UDP 14550 или USB/Serial tty).
3. Загружаются и рендерятся базовые QML-экраны (загрузка `FlyView` в состоянии "No vehicle").

*Синхронизация:* На данном этапе синхронизация идет только с локальным системным хранилищем настроек (QSettings / JSON).

---

## 2. Подключение к устройству

**Сценарий:** БПЛА включен и отправляет радиопакеты, GCS получает первый Heartbeat.

- **Участвующие модули:** `Comms` (физический транспорт), `MAVLink Protocol`, `MultiVehicleManager`, `InitialConnectStateMachine`.
- **Какие данные передаются:** `HEARTBEAT` (содержит system ID, component ID, autopilot type, base mode).
- **События-триггеры:** `LinkManager` парсит байты и успешно декодирует пакет `HEARTBEAT`.
- **Изменяемые состояния:** `Disconnected` -> `Connecting` (Запуск State Machine).

**Step-by-step последовательность:**
1. `LinkManager` оповещает `MultiVehicleManager` о новом источнике MAVLink.
2. `MultiVehicleManager` сканирует тип автопилота из пакета (например, MAV_AUTOPILOT_PX4).
3. Инстанцируется новый класс `Vehicle`. Этому классу назначается соответствующий `FirmwarePlugin`.
4. Для предотвращения гонок и перегрузки канала запускается `InitialConnectStateMachine`.

*Синхронизация:* QGC привязывает уникальный `System ID` к сущности Vehicle. UI подписывается на этот инстанс через биндинг `QGC.multiVehicleManager.activeVehicle`.

---

## 3. Инициализация данных (Parameters & Telemetry)

**Сценарий:** Станция GCS синхронизирует кэш параметров и загружает конфигурацию подключенного устройства.

- **Участвующие модули:** `InitialConnectStateMachine`, `ParameterManager`, `FactSystem`, `MissionManager`.
- **Какие данные передаются:** 
  - MAVLink-пакет `PARAM_REQUEST_LIST` (от QGC).
  - Поток MAVLink-пакетов `PARAM_VALUE` (от БПЛА).
  - Телеметрия: `SYS_STATUS`, `GPS_RAW_INT`, `ATTITUDE` (от БПЛА).
- **События-триггеры:** Успешный Heartbeat инициализирует стейтмашину загрузки параметров.
- **Изменяемые состояния:** `Connecting` -> `Syncing Parameters`.

**Step-by-step последовательность:**
1. `ParameterManager` запрашивает полный список параметров у БПЛА.
2. Прилетают сотни асинхронных ответов `PARAM_VALUE`. По мере поступления, они маппятся в классы `Fact` (устанавливается `value` и метаданные).
3. Если загрузка прерывается — происходит перезапрос пропавших пакетов `PARAM_REQUEST_READ`.
4. Параллельно `MissionManager` запрашивает текущую миссию (если она уже загружена на борт).
5. После скачивания всех фактов `InitialConnectStateMachine` переходит в финал.

*Синхронизация:* Существует жесткий механизм ожидания ACK-ответов от устройства. Состояние `Fact` в памяти QGC становится зеркалом состояния памяти контроллера аппарата.

---

## 4. Ready State (Состояние Готовности)

**Сценарий:** Параметры скачаны. Система готова к работе. Телеметрия обновляет дашборды.

- **Участвующие модули:** `Vehicle`, `FactSystem`, `VideoManager`, `UX (FlyView/PlanView)`.
- **Какие данные передаются:** Высокочастотные потоки MAVLink (pitch, roll, yaw, высоты, GPS, заряд батареи). Видеострим (RTP/RTSP) заводится в пайплайн виджета карты.
- **События-триггеры:** Окончание работы `InitialConnectStateMachine`. Флаг `initialConnectComplete` = True.
- **Изменяемые состояния:** `Syncing Parameters` -> `ReadyToFly` / `Active`.

**Step-by-step последовательность:**
1. QML-компоненты перестают показывать Loading. Оживают датчики и авиагоризонт (биндинг к факту `vehicle.pitch` / `vehicle.roll`).
2. В фоне происходит периодическое (например, 5 Гц) обновление проперти `Fact` на основе входящих данных MAVLink, что триггерит QML event loop посредством `valueChanged()`.

*Синхронизация:* На этом этапе QGC выступает пассивным потребителем (Listener) телеметрии. Обновления происходят однонаправленно (БПЛА -> QGC).

---

## 5. Выполнение миссии

**Сценарий:** Пользователь (или скрипт) загружает миссию и командует "Взлёт/Auto".

- **Участвующие модули:** `MissionManager`, `PlanView`, `Vehicle`, `FirmwarePlugin`.
- **Какие данные передаются:** `MISSION_ITEM` / `MISSION_ITEM_INT` (загрузка точек), команда `MAV_CMD_MISSION_START` или смена режима на `AUTO`. События `MISSION_ITEM_REACHED`.
- **События-триггеры:** Нажатие кнопок на UI ("Upload Required", "Start Mission").
- **Изменяемые состояния:** Режим полета аппарата меняется (например, `Stabilized` -> `Auto`), статус `Armed` становится True.

**Step-by-step последовательность:**
1. **Планирование:** В `PlanView` создаются путевые точки.
2. **Upload:** При запросе `MissionManager` передает счетчик точек (`MISSION_COUNT`), затем БПЛА запрашивает каждую точку по очереди. QGC отвечает `MISSION_ITEM`.
3. Завершается Upload (получен `MISSION_ACK`).
4. **Старт:** Пользователь делает Swipe для старта (свайп защищает от случайного нажатия).
5. QGC отправляет MAVLink команду на arming `MAV_CMD_COMPONENT_ARM_DISARM` и команду смены режима (`setFlightMode`).
6. **Выполнение:** Дрон взлетает. QGC слушает `MISSION_ITEM_REACHED` и перекрашивает пройденные Waypoints на 3D-карте в другой цвет.

*Синхронизация:* Сложная двунаправленная синхронизация. QGC контролирует (отправляет команды, требующие ACK-подтверждения), после чего переходит в режим ожидания исполнения, опираясь на высокочастотную телеметрию о глобальной позиции.

---

## 6. Завершение

**Сценарий:** Миссия выполнена. Дрон садится, дизармится и отключается.

- **Участвующие модули:** `Vehicle`, `LinkManager`, `MultiVehicleManager`.
- **Какие данные передаются:** MAVLink пакет с изменением `base_mode` (Disarm). Прекращение потока `HEARTBEAT`.
- **События-триггеры:** Посадка (событие от автопилота). Отключение питания БПЛА. Тайм-аут сокета в `LinkManager` (превышен лимит неполучения Heartbeat).
- **Изменяемые состояния:** `Active` -> `Link Lost` / `Disconnected`.

**Step-by-step последовательность:**
1. После достижения финальной точки, автопилот приземляет судно и снимает питание с моторов (Disarm).
2. UI-биндинг `vehicle.armed` становится False (кнопки `Start Mission` могут разблокироваться для следующей итерации).
3. При отключении питания (извлечение АКБ), сокет перестает получать heartbeat.
4. `LinkManager` определяет обрыв связи, вызывает процедуру закрытия транспортного канала.
5. Вызывается сигнал `vehicleLinkManager->closeVehicle()`.
6. `MultiVehicleManager` удаляет инстанс памяти (удаляются `MissionManager`, `FactSystem` и объект `Vehicle` для этого канала).
7. UI сбрасывается к начальному состоянию.

*Синхронизация:* Очистка данных из памяти QGC (разрыв Q_PROPERTY биндингов с QML-контекстом). Сборка локального кэша и остановка всех State-машин.

# 📊 Отчёт по анализу репозитория QGroundControl

> **Репозиторий:** qgroundcontrol-master
> **Дата анализа:** 2 Апреля 2026
> **Тип архитектуры:** Monolith (Qt C++ Backend / QML Frontend)
> **Статус гипотезы:** Подтверждена. Это крупное кроссплатформенное приложение-монолит для управления БПЛА, использующее Qt6 и протокол MAVLink.

---

## 🧭 Executive Summary
QGroundControl (QGC) — это кроссплатформенная (Windows, macOS, Linux, iOS, Android) станция наземного управления (GCS) для беспилотников. Проект имеет устоявшуюся архитектуру, строго завязанную на фреймворк Qt. Бэкенд написан на C++, а интерфейс — на QML. Система использует "Fact System" для абстракции состояния дрона и параметров и "FirmwarePlugin" для прозрачной поддержки различных прошивок (PX4, ArduPilot). 

---

## 🗺 Карта проекта

### 1. Общий стек технологий
- **Языки:** C++ (backend), QML/JS (frontend), Python (тулинг)
- **Фреймворк:** Qt 6 (Core, Quick, QML, Location, Multimedia, HttpServer)
- **Сборка:** CMake + Ninja
- **Линтеры/Форматтеры:** clang-format, qmlformat, qmllint, ruff (для python тулзов), pre-commit

### 2. Структура репозитория
- `src/` — исходный код C++ и QML модулей.
- `src/ui/` и `src/...View/` — QML интерфейсы (FlyView, PlanView, AnalyzeView).
- `tools/` — Python скрипты для CI/CD проверки и подготовки окружения.
- `resources/` — Графические и системные ресурсы, `*.qrc`.
- `cmake/` — Модули для системы сборки CMake.
- `.github/` — Github Actions (конфиги пайплайнов) и скрипты CI.

### 3. Ключевые модули и их назначение
- **Vehicle:** (`src/Vehicle`) Логическое представление дрона (MultiVehicleManager, Vehicle.cc).
- **FactSystem:** (`src/FactSystem`) Центральная система MVC. Маппит настройки и параметры машины (`Fact`) в QML для биндинга.
- **FirmwarePlugin:** (`src/FirmwarePlugin`) Абстрактный слой поверх автопилотов (PX4, ArduPilot), скрывающий детали реализации конкретных команд.
- **MAVLink:** (`src/MAVLink`) Обработчики протокола MAVLink для связи с БПЛА.
- **MissionManager:** (`src/MissionManager`) Логика планирования миссий, путевых точек, геозон (GeoFence) и точек возврата (RallyPoints).
- **Settings:** (`src/Settings`) Настройки приложения QGC.
- **VideoManager:** (`src/VideoManager`) Прием и стриминг видео с бортовой камеры с помощью GStreamer.

### 4. Основные зависимости и точки интеграции
- **Qt6 Plugins:** QSQLiteDriverPlugin (кэш карт), QtLocation (карты).
- **GStreamer:** Аппаратное/программное декодирование видеостримов (через UDP/RTSP).
- **MAVLink протокол:** Основная интеграция взаимодействия с дронами с использованием MAV_CMD команд.

### 5. Конфигурационные и инфраструктурные элементы
- Конфиги сборки: `CMakeLists.txt` верхнего и внутреннего уровней.
- CI/CD: GitHub Actions (сборки под iOS, Android, Linux, macOS, Win), тестируемые через `pytest` вспомогательные python-модули в `.github/scripts/`.
- Поддержка конфигурации модулей осуществляется через файлы `.SettingsGroup.json` в `src/Settings/`.

## Часть 1 — 🛠 Технический слой (Архитектура)

### 1.1 Базовый стек
| Слой | Технология | Версия | Роль |
|---|---|---|---|
| Язык | C++, QML/JS | C++20 | Вся бизнес-логика и интерфейс |
| Фреймворк | Qt | Qt 6.x | Графический бекенд, сетевая абстракция, QML |
| Сборка | CMake | 3.25+ | Настройка конфигураций для различных ОС и кросс-компиляции (Android/iOS) |
| Видео | GStreamer | - | Обработка видеопотока с борта |

### 1.2 Архитектурный подход
**MVC (Модель-Представление-Контроллер) с реактивным биндингом QML.**
Система использует паттерн "Fact System":
- **Model:** C++ классы (`Fact`, `FactGroup`, `Vehicle`).
- **View:** QML файлы (`FlyView`, `PlanView`), которые напрямую биндятся к свойствам C++ посредством мета-системы Qt (`Q_PROPERTY(FactGroup* vehicle READ vehicleFactGroup)`).
- **Controller:** Механизмы `FirmwarePlugin`, `ParameterManager`, транслирующие действия UI в команды микроконтроллера.

### 1.3 Интеграции, внешние зависимости и API
- В коде нет традиционных REST-контроллеров или API шлюзов, потому что QGC — это thick-клиент.
- Главный **"API" — это шина MAVLink.** Доступ и обмен данными с БПЛА идет через `mavlink_message_t`. Реализован MAVLink routing / logging (`QGCMAVLink.h`, `MAVLinkLogManager.cc`).
- Взаимодействие с внешним миром также идет при обновлении прошивок (скачивание файлов) и карт (Tiles API-провайдеров через `QtLocation`).
- Хранилищем настроек выступает внутренний формат конфигурации (в `src/Settings` описан json-метаданными, а сам стор работает через QSettings) и локальная QSQLite для тайловой памяти карт (`QGCTileCacheDatabase`).

### 1.4 Риски, ограничения и технические особенности
- Вся система крайне чувствительна к версии Qt. Наблюдается масштабный переход на Qt6.
- Приложение работает как state-machine (подключение порта -> MAVLink heartbeat -> получение параметров -> Ready). В случае рассинхронизации MAVLink-пакетов логика может зависнуть на этапе `InitialConnectStateMachine`.
- Существует жесткая привязка логики UI (размеры шрифтов, палитры) к глобальным стилям (`QGCPalette`).

## Часть 2 — 💼 Продуктовый слой (Бизнес-логика)

### 2.1 Основные функции системы
QGroundControl служит единым командным пунктом для настройки автопилотов (Airframe, Sensors, Radio calibration) и непосредственного управления миссиями (Waypoints, Return-To-Launch). 

### 2.2 Бизнес-сущности (Data Model)
- **Vehicle:** Текущее подключенное устройство или несколько устройств (`MultiVehicleManager`). Отслеживает статус `armed/disarmed`, `flightMode`, заряд `batteries` и телеметрию.
- **Fact / FactMetaData:** Переменная настроек аппарата. У переменной есть диапазоны `min`/`max`, `defaultValue` и `units` (извлекается из metadata).
- **Plan / Mission:** Список действий для дрона. Хранится локально на приложении и синхронизируется на борт последовательно с помощью `MissionManager` и `TrajectoryPoints`.
- **SettingsGroup:** Параметры самого приложения QGroundControl (карты, авто-подключение, джойстик).

### 2.3 Роли пользователей
В коде нет классических RBAC/авторизационных модулей, так как это Desktop/Mobile утилита для личного или корпоративного использования. Существующие пользователи разделяются логически своими действиями (не системно):
- **Оператор/Пилот:** Наблюдатель за полетом (использует экран `FlyView`), управление камерой, джойстиком. Выполняет `sendMavCommand(MAV_CMD_NAV_TAKEOFF)`.
- **Планировщик:** Создание автоматических миссий (использует `PlanView`), расстановка путевых точек.
- **Инженер по настройке:** Калибровка сенсоров (Accelerometers, Gyro, Compass), загрузка свежей прошивки в `VehicleSetup`.

### 2.4 Ключевые сценарии использования
- **Настройка дрона (Vehicle Setup):** Подключение через `Comms` (Serial, UDP, TCP). Срабатывание `InitialConnectStateMachine`. Скачивание текущих параметров через `ParameterManager` (FactSystem). Обновление UI-форм с доступными настройками.
- **Планирование (Mission Planning):** Оператор в QML (`PlanView.qml`) добавляет точку. Бэкенд обновляет `QmlObjectListModel`. При нажатии 'Upload', `MissionManager` загружает список MAVLink-команд на борт с ожиданием ACK ответа.
- **Управление полетом (Flight Control):** Вкладка `FlyView`. QGC получает потоковую телеметрию от Vehicle. Компоненты UI (например `InstrumentValueIcons`) изменяются реактивно на основе свойств `FactGroup`. Можно использовать джойстик (отправка `MANUAL_CONTROL` MAVLink сообщений).
- **Видео Трансляция:** Получение потока от камеры в `VideoManager` через `VideoReceiver` GStreamer, отображение поверх интерфейса карты.

## ❓ 3. Допущения, неизвестные зоны и открытые вопросы

- [ ] **Сторонние плагины и Custom Builds:** Видно, что в `CMakeLists.txt` предусмотрена логика интеграции `custom/` проприетарных кастомных сборок QGC (для коммерческих клиентов). Полное влияние кастомных плагинов по коду не до конца отслежено.
- [ ] **Поддержка UAVCAN:** В коде (`Vehicle`) есть упоминания `startUAVCANBusConfig`. В какой степени реализовано конфигурирование аппаратуры по интерфейсу CAN напрямую из QGC — неизвестно без глубокого погружения.
- [ ] **Реализация VideoSink (Отображение GStreamer):** Раньше интеграция GStreamer с Qt могла быть нестабильной (Qt5). Видно наличие модулей `gstqml6gl`, требуется верификация их работы на всех мобильных платформах (Android/iOS).
- [ ] **Тестирование:** В репозитории есть папка `test`, но в `CMakeLists.txt` по умолчанию `QGC_BUILD_TESTING = OFF`. Вопрос: насколько регулярно прогоняются интеграционные тесты с симулятором (например, `SITL` Gazebo/Ignition). Наверняка они вынесены в `playwright_runner.py` или CI инструменты.
